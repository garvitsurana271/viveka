# Viveka — Evaluation Methodology & Results

Viveka is evaluated on **[AVeriTeC](https://fever.ai/dataset/averitec.html)**, the
standard academic benchmark for real-world claim verification: 4,568 claims drawn
from 50 fact-checking organisations, each labelled **Supported / Refuted /
Conflicting Evidence-Cherrypicking / Not Enough Evidence (NEI)**. We score on the
**dev split (500 claims)** and draw few-shot exemplars from **train (3,068)**.

This document is the honest record of *how* we measure and *what we found* — including
a result that did **not** work. The whole point of building a benchmark harness was
to stop shipping on intuition.

## Why these metrics

Dev is **61% Refuted**, so a model that blindly answers "Refuted" scores 0.61
accuracy while being useless. We therefore report, alongside accuracy:

- **Macro-F1** — the unweighted mean of per-class F1. This is the honest headline:
  the lazy always-Refuted classifier scores ~0.19 here.
- **Per-class precision / recall / F1** and the **full confusion matrix** — so a
  weakness in one class (ours is *Conflicting*) is visible, not hidden in an average.
- **Abstention rate** — how often the system fell back to NEI.

## The two evaluation tracks

A single number hides whether a miss is a *reasoning* failure or a *retrieval*
failure. So we score two tracks:

1. **Gold-evidence reasoning (Track A).** Feed the model AVeriTeC's own
   researched questions+answers and ask only for the label. This isolates
   *veracity reasoning* — given good evidence, can it aggregate to the right
   verdict? One Gemma call per claim, fully reproducible.

   > **$0 deviation, disclosed.** AVeriTeC's full "knowledge store" is hundreds of
   > GB (≈1,000 scraped URLs per claim) and cannot be hosted for free. Track A uses
   > the dataset's *gold* QA as the evidence source instead — a standard
   > "veracity-prediction-given-evidence" setup. It measures reasoning, not retrieval.

2. **Live end-to-end (Track B).** Run Viveka's real pipeline on the bare claim —
   `analyze → bounded agentic multi-hop retrieval (ddgs + Wikipedia + Google Fact
   Check) → verdict`. This is what a real user actually gets.

## Methods compared (the ablation)

| Tag | What it is |
|---|---|
| `gold` | Single Gemma pass over gold evidence (the reasoning baseline). |
| `gold +fewshot` | Same, with train exemplars over-weighting *Conflicting*. |
| `golddebate` | **Multi-agent debate** over gold evidence: 3 independent lenses (skeptic / literalist / charitable) + a judge, where a supported-vs-refuted split is read as *Conflicting*. |
| `P2` (Likert-softmax) | Rate all four labels 1-5, softmax to a distribution, abstain on a domain-aware NEI margin instead of a single self-reported confidence. |
| `P2P3` (Likert + dynamic exemplars) | **P2 plus** seeding the prompt with the K nearest already-checked claims (BM25). **This is what the product ships.** |
| `agentic` | Live end-to-end with the bounded multi-hop loop. |
| `debate` | Live end-to-end + the debate panel. |

## Results (AVeriTeC dev, n=100, gold-evidence track)

_Headline metric is macro-F1. Table auto-generated by `eval/compare.py`._

**Three caveats, stated up front.** (1) This is a **balanced 25-per-class slice**, balanced
deliberately so macro-F1 is not inflated by AVeriTeC's natural ~61%-Refuted majority.
(2) It is **gold-evidence, label-only** verdict quality, **not** the official Ev2R-conditioned
AVeriTeC score (SOTA around 0.48). (3) The shipped recipe still **abstains on ~36%** of cases
*even with perfect gold evidence* (the high-stakes confidence floor declining to commit); the
live end-to-end number is **0.398** (n=40 balanced) with ~15% abstention. n is small; read the
decimals as directional.

| Method | Accuracy | Macro-F1 | Conflicting recall | Note |
|---|---|---|---|---|
| **`P2P3`** (Likert + dynamic exemplars) | **0.730** | **0.733** | 0.60 | **best — the recipe the product ships**; +0.097 over baseline, Conflicting·F1 0.56→0.73 |
| `P2` (Likert-softmax alone) | 0.700 | 0.700 | 0.52 | fixed the over-abstention; +0.064 over baseline on its own |
| `gold` (baseline) | 0.640 | 0.636 | 0.40 | strong on Sup/Ref, retreats to NEI on Conflicting |
| `golddebate` | 0.630 | 0.625 | 0.56 | lifts Refuted (0.64→0.73) & Conflicting (0.56→0.62), but the skeptic lens over-doubts true claims (Supported 0.71→0.54) — a wash on label accuracy |
| `gold +fewshot` | 0.620 | 0.623 | 0.52 | **fixed Conflicting recall but net-negative** — over-taught the pattern, *not shipped* |

### Debate didn't win on accuracy — but it delivered a calibrated uncertainty signal

The accuracy win is **P2P3** (Likert-softmax + dynamic exemplars), and that is what
the product reasons with. Debate *tied* the baseline on label accuracy — so we did
not ship it for accuracy, and we say so. But the debate panel surfaced something a
single pass cannot: **panel agreement tracks correctness.**

- When the 3 lenses **agree** (77/100 claims): **70%** accurate.
- When they **split** (23/100 claims): **39%** accurate, and splits are enriched
  for genuinely-Conflicting claims (43% vs the 25% base rate).

Compare that to the model's *self-reported* confidence — ECE **0.327**, flat at
~98% whether right or wrong. So the engine's `product` mode abstains to a human on
**panel disagreement**, a signal that actually separates 70% from 39% — not on a
confidence number the model fabricates. That is the responsible-AI payoff, and it
is only visible because the harness measured it.

### The negative result, kept on the record

Static few-shot did exactly what it was designed to (Conflicting recall
0.40 → 0.52) yet **lowered macro-F1** (0.636 → 0.623): Supported claims began
leaking into Conflicting. Intuition says "few-shot always helps"; the harness
proved otherwise on this task. We did not ship it as a win. The better lever is a
*mechanism* (debate), not more examples.

## Calibration

`eval/calibration.py` bins predictions by stated confidence and compares each bin
to its empirical accuracy, reporting **ECE (Expected Calibration Error)** and the
confidence threshold at which product mode should abstain to a human. A verdict
that says "90% sure" should be right ~90% of the time; we measure whether it is.

The model's *self-reported* confidence fails this badly: **ECE 0.327**, pinned near
~98% whether right or wrong. The product does **not** display that number. It shows
the **Likert-softmax** probability of the chosen label at a measured temperature,
which lands **ECE ~0.21** — a clear verdict reads ~60-67%, roughly how often it is
actually right at that level. The displayed confidence is the honest one, not the
flattering one. (The deeper, better-separated signal is **panel agreement**, above:
70% accurate when the lenses agree vs 39% when they split.)

## Two decision modes, one engine

- **benchmark mode** commits to one of the four labels (for scoring).
- **product mode** keeps the 5-label taxonomy and **abstains to a human** below the
  calibrated confidence threshold — the responsible-AI behaviour. Safe abstention
  helps the product and *hurts* the benchmark (an abstention on a Refuted claim
  scores wrong), which is exactly why the two modes are separated.

## Reproducing

```bash
# from backend/, with the venv python; throttled to stay under Gemma's 15 RPM
python eval/averitec_runner.py --split dev --n 100 --track gold        --workers 4
python eval/averitec_runner.py --split dev --n 100 --track gold        --workers 4 --fewshot
python eval/averitec_runner.py --split dev --n 100 --track golddebate  --workers 4
python eval/averitec_runner.py --split dev --n 40  --track agentic     --workers 4
python eval/compare.py --md eval/reports/RESULTS.md   # regenerate the table
python eval/calibration.py eval/reports/baseline_dev_n100_gold.json
```

Every Gemma call is disk-cached by prompt hash (`eval/cache/`), so re-runs only
pay for prompts that actually changed — a full re-score after a code tweak is
near-free, which is what makes iterating under a 1,500/day quota possible.
